# MANUAL_STEPS — что нужно сделать руками

Этот файл собирает все действия, которые нужно выполнить вручную в внешних
сервисах (Supabase Dashboard, Google Cloud Console и т.д.) — Claude Code не
имеет туда доступа. Пункты добавляются по мере готовности модулей, старые не
удаляются.

## Модуль 0: фундамент

- Применить `supabase/migrations/0001_init.sql` через Supabase SQL Editor.
- Выполнить `supabase/seed.sql` (города Ингушетии, 19 категорий).
- Проверить `.env.local` в `apps/web`: `NEXT_PUBLIC_SUPABASE_URL`,
  `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`.

## Модуль 1: Auth и профиль (email + Google)

- Применить `supabase/migrations/0002_auth_profile.sql` (bucket `avatars`,
  синхронизация подтверждения email).
- Применить `supabase/migrations/0003_google_oauth.sql` (обновление
  `handle_new_user()` для аватара из Google).
- **Authentication → URL Configuration**: Site URL `http://localhost:3000`
  (после деплоя — реальный домен), Redirect URL
  `http://localhost:3000/auth/confirm`.
- **Authentication → Providers → Email**: включить Confirm email.
- Google OAuth: создать проект в Google Cloud Console → OAuth consent screen
  → Credentials → OAuth client ID (Web application) → добавить в Authorized
  redirect URIs адрес обратного вызова Supabase
  (`https://<project>.supabase.co/auth/v1/callback`) → вставить Client
  ID/Secret в Supabase **Authentication → Providers → Google**.
  Подробности — в [README.md](README.md#вход-через-google).

## Модуль 2: Admin CMS (ядро)

- Применить `supabase/migrations/0004_admin_cms.sql` (таблицы `banners`,
  `admin_audit_log`, bucket `site-assets`).
- Назначить себя администратором вручную через SQL Editor:
  ```sql
  insert into public.user_roles (user_id, role_id)
  select p.id, r.id
  from public.profiles p
  join auth.users u on u.id = p.id
  join public.roles r on r.code = 'admin'
  where u.email = 'ваш-email@example.com'
  on conflict (user_id, role_id) do nothing;
  ```
- Проверить в браузере: ссылка "Админка" в шапке, разделы Категории/Настройки
  сайта/Баннеры.

## Модуль 3: Объявления

- Применить `supabase/migrations/0005_listings.sql` через SQL Editor. Создаёт
  таблицы `listings`, `listing_images`, `listing_videos`, `listing_attributes`,
  `complaints` и bucket `listing-media` для фото.
- Если хотите проверить модерацию: назначьте себе роль `moderator` (или
  используйте уже назначенную `admin`) — так же, как в шаге "Назначить себя
  администратором" из Модуля 2, но `where r.code = 'moderator'`.
- Проверить в браузере:
  1. Войдите под обычным аккаунтом → в шапке должна появиться кнопка
     "+ Подать объявление".
  2. Заполните форму → попадаете на страницу редактирования черновика.
  3. Загрузите хотя бы одно фото → нажмите "Отправить на модерацию".
  4. Зайдите в Админку → **Модерация объявлений** под аккаунтом
     admin/moderator → одобрите или отклоните объявление.
  5. После одобрения объявление должно открываться по адресу
     `/listings/<slug>` и быть видно без авторизации.
  6. На странице объявления (не своего) — кнопка "Пожаловаться", жалоба
     должна появиться в Админке → **Жалобы**.
  7. В "Мои объявления" проверьте кнопки "Продано" / "В архив" / "Удалить"
     (для черновика).

### Известные упрощения этого модуля (не баг, осознанный выбор объёма)

- Видео к объявлению — только ссылка на внешний хостинг (YouTube/VK Видео),
  без загрузки файла в Storage. Полноценная загрузка видео — можно добавить
  отдельной задачей позже, если понадобится.
- У категорий пока нет ни одного заполненного набора атрибутов
  (`category_attributes` пуст) — форма объявления уже умеет их показывать
  динамически, но проверить это можно только после того, как через
  Админку → Категории (или SQL) будут добавлены атрибуты для какой-то
  категории. Управление атрибутами из самой админ-панели категорий пока не
  реализовано — его можно добавить отдельной задачей.
- Удаление фото из объявления удаляет только запись в БД — сам файл в
  Supabase Storage не удаляется (не критично для MVP, но занимает место).
- Счётчик просмотров и защита от накрутки — в Модуле 6 ("Избранное и
  просмотры"), в этом модуле не добавлялись.

## Модуль 4: Поиск и фильтры

- Применить `supabase/migrations/0006_search_and_filters.sql` через SQL
  Editor. Добавляет полнотекстовый поиск (`search_vector`, конфигурация
  `russian`) и денормализованные поля `cover_image_url`/`has_video` на
  `listings` (обновляются триггерами автоматически при добавлении/удалении
  фото и видео — руками ничего заполнять не нужно, но именно поэтому
  применить миграцию нужно один раз явно).
- Проверить в браузере:
  1. На главной клик по любой категории → открывается `/category/<slug>`
     со списком объявлений этой категории (и подкатегорий), фильтрами слева
     (город/цена/состояние/фото/видео/VIP) и сортировкой.
  2. Поле поиска в шапке сайта → ведёт на `/search?q=...` с теми же
     фильтрами.
  3. После того как у вас появится хотя бы одно опубликованное объявление
     (Модуль 3) — проверьте, что оно находится по подстроке из заголовка/
     описания и что фильтр "С фото"/по городу/по цене отсекает лишнее.
  4. Проверьте пагинацию (появляется, если объявлений в выдаче больше 24).

### Известные упрощения этого модуля

- Поиск — по `title`/`description` через `to_tsvector('russian', ...)`.
  Показ результатов по подстроке в характеристиках (`listing_attributes`)
  не реализован — можно добавить отдельной задачей, если понадобится.
- Фильтр "Только VIP" уже работает технически, но включать объявления как
  VIP пока негде в интерфейсе — колонка `is_vip` появится в UI в Модуле 10
  (монетизация).

## Модуль 5: Карта

- Получить бесплатный ключ **JavaScript API и HTTP Геокодер** на
  developer.tech.yandex.ru (нужен обычный Яндекс-аккаунт — зарегистрировать
  проект в кабинете разработчика придётся вам, я туда доступа не имею).
- Вставить его в `apps/web/.env.local`:
  `NEXT_PUBLIC_YANDEX_MAPS_API_KEY=ваш_ключ`.
- Без ключа сайт не ломается — везде, где должна быть карта, показывается
  текстовая заглушка "Карта временно недоступна". Это уже проверено.
- После того как вставите ключ, проверить в браузере:
  1. `pnpm dev`, откройте `/map` — должна открыться карта Ингушетии с
     точками опубликованных объявлений (клик по точке ведёт на объявление).
  2. На странице объявления (Модуль 3) — под характеристиками должна
     появиться мини-карта с местоположением (либо точным, либо по центру
     населённого пункта, если точку не указывали).
  3. В форме подачи/редактирования объявления — кнопка "Указать точку на
     карте", клик по карте ставит красную метку и сохраняет `lat`/`lng`.

### Известные упрощения этого модуля

- Координаты объявления — необязательные и выставляются только вручную
  кликом по карте в форме подачи. Автоматического геокодирования введённого
  текстового адреса (`address_text` → координаты) нет — можно добавить
  отдельной задачей, если понадобится точность по адресу, а не только по
  клику.
- Компонент карты (`components/map/ListingsMap.tsx`) — единственное место в
  коде, завязанное на Яндекс.Карты; в БД координаты хранятся
  провайдер-независимо, поэтому смена провайдера в будущем не потребует
  миграций.
- Без API-ключа (до тех пор, пока не выполните шаг выше) везде вместо карты
  — текстовая заглушка; это ожидаемое поведение, не баг.

## Пивот: ING MARKET → Bazar

- Применить `supabase/migrations/0007_rebrand_bazar.sql` — обновляет
  название сайта в уже существующей записи `site_settings` (иначе в шапке

- Применить `supabase/migrations/0008_update_geo_data.sql` — актуализирует
  справочник районов/городов:
  - добавляет городской округ **Карабулак** (его не было вообще) и ~35 сёл,
    которых не хватало;
  - переименовывает городские округа в официальную форму ("Городской округ
    город Назрань" вместо "Городской округ Назрань" и т.п.);
  - чинит баг старого seed'а: станица "Орджоникидзевская" и город "Сунжа" —
    это одно и то же место под старым/новым названием, миграция переносит

    дубликат;
  - добавляет тип "станица" (Нестеровская, Троицкая, Вознесенская, Мужичи) —
    раньше схема такой тип не allow'ила;
  - добавляет `unique (name)` на `regions`/`cities`, чтобы такой баг
    (одно место — две записи под разными именами) не повторился.
- Проверить в браузере после применения обеих миграций:
  1. Название сайта в шапке — "Bazar".
  2. В форме подачи объявления (Модуль 3) в списке городов должен появиться
     **Карабулак** и заметно больше сёл, чем раньше.
  3. Если у вас уже было объявление или профиль с городом "Орджоникидзевская"
     — после миграции он должен показывать город "Сунжа" (проверить, что
     ничего не отвалилось).

### Точность данных — не проверено сверх источника, указанного пользователем

Список городов/районов и их типы взяты из брифа пользователя (источник —
geoadm.com/ОКТМО), сам список не перепроверялся мной по независимому
источнику. Координаты (`lat`/`lng`) для новых сёл, особенно в горном
Джейрахском районе, — грубые приближения "по центру примерно там", не по
точным данным. Для MVP (список в фильтре города, точка на карте "где-то
рядом") этого достаточно; если понадобится точная привязка — стоит
перепроверить координаты перед тем, как полагаться на них.

## Контакт с продавцом, избранное, "поднятие в списке"

Вместо полноценного чата — как договорились при пивоте на Bazar.

- Применить `supabase/migrations/0009_contact_favorites_boost.sql` — создаёт
  таблицы `favorites` и `listing_comments`, добавляет `listings.boosted_at` и
  `listings.sort_priority` (используется сортировкой "Сначала новые" — см.
  `lib/listings/query.ts`).
- Ничего дополнительно заводить не нужно (нет внешних сервисов/ключей) —
  телефон продавца берётся из уже существующего `profiles.phone`.
- Проверить в браузере после применения миграции:
  1. На странице объявления — телефон продавца скрыт за кнопкой "Показать
     телефон", после клика становится кликабельной `tel:`-ссылкой.
  2. Там же — кнопка "В избранное"/"В избранном" (нужно быть авторизованным);
     раздел **Избранное** в личном кабинете должен показывать добавленные
     объявления.
  3. Внизу страницы объявления — форма комментария (для авторизованных) и
     список уже оставленных комментариев; автор комментария и владелец
     объявления могут его удалить.
  4. В "Мои объявления" на опубликованном объявлении — кнопка "Поднять в
     списке"; должна работать не чаще раза в 24 часа (повторный клик раньше
     — ошибка с указанием, сколько часов осталось).
  5. В Админке → Модерация объявлений — чекбокс "Отметить VIP" рядом с
     кнопкой "Одобрить".

### Известные ограничения этого шага

- Кнопка "В избранное" сейчас есть только на странице объявления, не на
  карточках в каталоге/поиске/категориях — сделано так ради скорости,
  можно добавить позже, если понадобится хартить прямо из списка.
- Отметить объявление VIP можно только в момент одобрения на модерации; для
  уже опубликованного объявления — только через SQL
  (`update public.listings set is_vip = true where slug = '...'`), отдельной
  админ-страницы со списком всех объявлений пока нет.
- Комментарии — плоский список без ответов/вложенности и без модерации
  (кроме права автора/владельца объявления удалить).
- "Поднятие в списке" — просто сортировочный трюк (сбрасывает
  `sort_priority` на "сейчас"), никакого визуального выделения в каталоге у
  поднятого объявления нет, кроме позиции в списке.

## Расширение категорий подразделами (по образцу berkat.ru)

- Применить `supabase/migrations/0010_expand_categories.sql` в Supabase SQL
  Editor. Добавляет подкатегории почти во все разделы (например, "Автомобили"
  → Легковые/Грузовики и спецтехника/Мототранспорт/...) и новый раздел
  **Сельское хозяйство** (Животные/Птицы/Корма/Растения/Пчёлы/
  Сельхозоборудование) — "Животные" перестаёт быть отдельным разделом и
  становится его подразделом, как на конкурентном сайте.
- На главной странице лимит категорий поднят с 12 до 24, чтобы новый раздел
  и остальные не обрезались.
- Проверить в браузере: на главной должен появиться раздел "Сельское
  хозяйство"; при заходе в любой крупный раздел (например "Автомобили" или
  "Услуги") сверху должны появиться кнопки-подразделы.
- Структура подкатегорий составлена по мотивам berkat.ru, но не 1:1 копия
  (некоторые их совсем узкие листья вроде "Сканеры"/"МФУ" не переносились
  ради разумного объёма) — если каких-то конкретных подразделов не хватает,
  их можно добавить через Админку → Категории (там уже есть выбор
  родительской категории) без новой миграции.

## Мега-меню категорий + точечные подразделы (маркетинговый проход)

- Применить `supabase/migrations/0011_category_gaps.sql` в Supabase SQL

  Стройматериалы → Двери и окна; Услуги → Ремонт бытовой техники.
- Новое в интерфейсе (без миграции, уже в коде):
  - В шапке сайта (все страницы, кроме мобильной ширины) появилась кнопка
    "Категории" — открывает мега-меню в два столбца: слева все разделы,
    справа подразделы того, что под курсором. На мобильной ширине — плоский

  - На главной странице карточки категорий при наведении показывают
    всплывающий список своих подразделов (только на ширине от `md` —
    на телефоне наведения нет, поведение осталось прежним: тап открывает
    раздел). Под названием раздела также показано число подразделов
    ("6 подразделов").
  - Оба места используют один и тот же запрос категорий
    (`apps/web/lib/categories.ts`), лимит в 24 карточки на главной убран —
    теперь показываются все активные разделы верхнего уровня.
- Проверить в браузере: навести на любую карточку раздела на главной — под
  ней должен появиться список подразделов; нажать "Категории" в шапке —
  должно открыться меню в два столбца, при наведении на пункт слева правая
  колонка должна меняться.

## Технический SEO-минимум

- Добавлены `/robots.txt` и `/sitemap.xml` (генерируются кодом, не файлы —
  `apps/web/app/robots.ts` и `apps/web/app/sitemap.ts`). Sitemap включает
  главную, `/rules`, все активные категории и до 5000 активных объявлений.
  Ничего вручную настраивать не нужно, но **после покупки домена** — см.
  ниже — стоит явно зарегистрировать sitemap в Яндекс.Вебмастере и Google
  Search Console (Вебмастер сам подхватит его и по `robots.txt`, но ручная
  отправка ускоряет первую индексацию).
- У страниц объявления и категории теперь свои `<title>`/описание вместо
  одного и того же на весь сайт; у объявления в превью-ссылке (когда её
  кидают в WhatsApp/Telegram/VK) подтягивается первая фотография.
- `/search` помечен `noindex` — это динамическая страница результатов,
  индексировать её страницам категорий незачем (дублирует контент).
  Сами страницы категорий и объявлений остаются индексируемыми.
- Добавлена страница `/rules` ("Правила размещения") — ссылка есть в футере
  на каждой странице.
- **Не сделано и почему:** кастомная OG-картинка (фирменная картинка-превью
  при расшаривании ссылки на сайт целиком, не на объявление) — код был
  написан через `next/og`, но упёрся в баг самого Next.js 15.0.8: генерация
  такой картинки ломает `next build` на Windows (issue vercel/next.js#77164,
  `path.join` некорректно собирает `file://`-путь на Windows). На Linux
  (там, где реально собирает Vercel) бага нет, но подтвердить это можно
  только реальным деплоем, а ломать рабочий билд ради превью-картинки не
  стоило — решение убрано. Если понадобится вернуть: собрать картинку через
  `next/og`, `ImageResponse`, без сборки на Windows-машине (или после
  апдейта Next.js, где баг может быть уже исправлен).
- **Домен.** Сайт пока живёт на `<ваш-проект>.vercel.app` — если ещё не
  куплен `.ru`/`.рф`-домен, самое время: Vercel → Settings → Domains после
  покупки у любого регистратора. Без своего домена смысла регистрироваться
  в Яндекс.Вебмастере почти нет — люди не запомнят и не введут вручную
  vercel-адрес.

## Публикация объявлений без модерации + логичное поле "Состояние"

- Применить `supabase/migrations/0012_publish_flow.sql` в Supabase SQL
  Editor. Без неё раздел "Категории" на сайте будет пустым (запрос просит
  колонку `show_condition`, которой ещё нет) — это ожидаемо, не баг.
- **Что изменилось в правилах публикации:** раньше объявление после
  заполнения формы уходило в очередь "на модерации" и ждало, пока
  админ/модератор его одобрит через `/admin/listings`. Теперь автор
  публикует объявление сам, сразу — кнопка называется "Опубликовать" и
  сразу выставляет статус "active". На уровне базы данных это разрешено
  триггером `enforce_listing_status` — единственное, что по-прежнему может
  выставить только модератор, это статус "Отклонено" (это вердикт по
  чужому объявлению, не то, что человек ставит сам себе).
  - Раздел "На модерации" в админке (`/admin/listings`) никуda не делся —
    просто в обычном сценарии он теперь почти всегда будет пустым. Он
    останется полезен, если модератор вручную отклонит объявление
    (`rejected`) — история отклонения видна автору при повторном
    редактировании.
- **Форма подачи объявления теперь в два по-настоящему понятных шага**
  ("Шаг 1 из 2" / "Шаг 2 из 2" видно на экране): на первом шаге —
  название, категория, цена, описание; на втором — фото и кнопка
  "Опубликовать". Раньше единственной кнопкой было "Создать черновик",
  что выглядело как тупик, а не как публикация.
- **Категория теперь выбирается в два уровня** (раздел → подраздел)
  вместо одного длинного списка с отступами — быстрее находить нужное,
  особенно на телефоне.
- **Поле "Состояние" (новое/б/у) показывается не везде.** У каждой
  категории есть флаг `show_condition` в базе — для недвижимости, услуг,
  вакансий, живых животных/растений и продажи бизнеса он выключен (см.
  список в самой миграции), поэтому у "Земельного участка" больше не
  будет спрашивать "новое или б/у". Если понадобится включить/выключить
  это поле для какой-то ещё категории — пока только через SQL
  (`update categories set show_condition = false where slug = '...'`),
  в интерфейсе админки такого переключателя нет.
- **Про жалобу "на модерации: 1, захожу — пусто":** проверил RLS и запросы
  админки — политики корректные, staff видит все статусы. Вероятная
  причина — кэш страницы `/admin/listings` в Next.js не обновлялся при
  отправке объявления (страница не была в списке `revalidatePath`).
  Добавил `revalidatePath("/admin/listings")` при публикации — если всё
  же повторится после этой миграции, пришлите скриншот, разберёмся
  дальше.

## Тип сделки для недвижимости (Продам/Сдам/Куплю/Сниму)

- Применить `supabase/migrations/0013_deal_type.sql` в Supabase SQL Editor.
- У недвижимости (и всех её подразделов — квартиры, дома, участки,
  коммерческая, гаражи) при подаче объявления теперь обязательно нужно
  выбрать тип сделки: **Продам** / **Сдам** / **Куплю** / **Сниму** — то,
  чего не хватало раньше (по аналогии с тем, как в "Работе" уже разделены
  "Вакансии" и "Ищу работу"). Для остальных категорий это поле не
  показывается вообще.
- Тип сделки виден бейджем на карточке объявления, на странице объявления
  и есть отдельным фильтром на странице поиска и категории.
- Как и с `show_condition`, включить это поле для ещё какой-то категории
  можно пока только через SQL (`update categories set show_deal_type = true
  where slug = '...'`).

## Порядок разделов и подразделов по реальному спросу

- Применить `supabase/migrations/0014_reorder_by_demand.sql` в Supabase SQL
  Editor.
- Что сделано: зашёл на berkat.ru и выписал число объявлений в каждом их
  разделе и подразделе — это реальный сигнал того, что востребовано в
  регионе. По нему переставил порядок и у нас — и разделы верхнего уровня
  (например, "Автомобили" и "Телефоны" теперь идут раньше "Хобби" и
  "Бизнеса"), и подразделы внутри каждого раздела (например, в "Сельском
  хозяйстве" подраздел "Животные" теперь идёт первым, как и должно быть —
  раньше он шёл последним из-за того, как переносился в отдельный раздел
  в старой миграции).
- Добавлены 3 подраздела, которых не хватало и которые реально
  востребованы у конкурента, но названы по-своему, не дословно:
  "Спутниковое телевидение" (Электроника), "Подарки и сувениры" (Дом и
  сад), "Продукты питания" (Сельское хозяйство).

  "Сейфы", "Сканеры" — единицы объявлений даже у них) — ради разумного
  объёма каталога, это уже было решение при первом расширении категорий
  (миграция 0010) и осталось в силе.
- Порядок между сайтами не идентичный специально — где спрос был не
  очевиден или структура категорий сильно отличается (например,
  "Услуги" у нас объединяют то, что у конкурента разложено на
  "Строительство" + "Услуги" отдельно), полагался на здравый смысл,
